# 예시 문서

norii로 열어 눈으로 확인하는 문서 모음이다. 각 파일은 하나의 기능 표면을 담당하며, 그 기능이 실제로 어떻게 보이고 어디서 무너지는지를 한 화면에 담는다.

빌드·테스트에 들어가지 않는다. 자동 테스트가 "동작한다"를 증명한다면, 이 문서들은 **"어떻게 보이는가"** 를 사람이 판정하게 한다 — 타이포·간격·스크롤·깜빡임처럼 통과/실패로 자를 수 없는 것들이다.

스크롤 동기화처럼 상호작용으로만 시연되는 기능은 전용 문서를 두지 않는다 — 위 예시들이 그 재료를 이미 제공한다(수식·다이어그램은 소스 몇 줄이 화면 한 뭉텅이가 되어 양쪽 높이가 크게 어긋난다).

## 생성 기준

생성 여부의 판단은 [작업 규칙 — 예시 문서](../.claude/rules/project-rules.md#예시-문서-examples)가 단일 출처다. 대부분의 기능은 대상이 아니며, 대상이어도 기존 파일에 절을 더하는 것이 우선이다. 이 문서는 목록과 작성 규칙만 소유한다.

## 작성 규칙

각 파일은 다음 셋을 모두 담는다. 정상 케이스만 담은 예시는 불완전하다 — 무너지는 자리까지 보여야 그 기능을 판정할 수 있다.

1. **평범한 사용** — 실제로 쓰게 될 모습.
2. **경계** — 폭을 넘는 표·수식·다이어그램(자기 안에서만 가로 스크롤되는지), 같은 것이 두 번 나오는 경우.
3. **의도적인 실패** — 문법이 틀린 수식·다이어그램. 앱이 깨지지 않고 **그 자리에만** 알리는지, 뒤따르는 내용이 온전한지 확인한다.

알려진 한계는 문서 안에 그대로 적는다(예: 각주를 눌러도 이동하지 않음). 예시 문서가 제품의 현재 상태를 정직하게 비추게 한다.

**포매터는 이 폴더를 건드리지 않는다**(`.oxfmtrc.json`의 `ignorePatterns`). 예시의 목적이 사용자가 실제로 쓰는 원문을 보여 주는 것이기 때문이다 — 포맷을 적용하면 `[X]`가 `[x]`로, `*기울임*`이 `_기울임_`으로 정규화되어 변형을 지원한다는 사실을 보여 줄 수 없다.
