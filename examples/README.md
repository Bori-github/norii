# 예시 문서

norii로 **열어서 눈으로 확인하는** 문서 모음이다. 각 파일은 하나의 기능 표면을 담당하며, 그 기능이 실제로 어떻게 보이고 어디서 무너지는지를 한 화면에서 보여 준다.

빌드·테스트에 들어가지 않는다. 자동 테스트가 "동작한다"를 증명한다면, 이 문서들은 **"어떻게 보이는가"** 를 사람이 판정하게 한다 — 타이포·간격·스크롤·깜빡임처럼 통과/실패로 못 자르는 것들이다.

## 언제 만드나

**만들 것인가 말 것인가의 판단은 [작업 규칙 — 예시 문서](../.claude/rules/project-rules.md#예시-문서-examples)가 단일 출처다.** 요약하면 대부분의 기능은 대상이 아니고, 대상이어도 기존 파일에 절을 더하는 것이 먼저다 — 새 파일은 마지막 수단이다.

여기(README)는 **무엇이 있는지**(위 목록)와 **어떻게 쓰는지**(아래)만 소유한다.

## 어떻게 쓰나

각 파일은 세 가지를 모두 담는다. 정상 케이스만 있는 예시는 절반짜리다 — 무너지는 자리를 봐야 그 기능을 안다고 할 수 있다.

**포매터는 이 폴더를 건드리지 않는다**(`.oxfmtrc.json`의 `ignorePatterns`). 예시의 존재 이유가 **사용자가 실제로 쓰는 원문을 보여 주는 것**이기 때문이다 — 포맷을 걸면 `[X]`가 `[x]`로, `*기울임*`이 `_기울임_`으로 정규화되어 "이 변형도 지원한다"를 보여 줄 수가 없다. 실제 사용자는 정규화된 마크다운을 쓰지 않는다.

1. **평범한 사용** — 실제로 쓰게 될 모습.
2. **경계** — 폭을 넘는 표·수식·다이어그램(자기 안에서만 가로 스크롤되는가), 같은 것이 두 번 나오는 경우.
3. **의도적인 실패** — 문법이 틀린 수식·다이어그램. 앱이 깨지지 않고 **그 자리에만** 알리는지, 뒤따르는 내용이 멀쩡한지 본다.

알려진 한계는 문서 안에 **그대로 적는다**(예: 각주를 눌러도 이동하지 않음). 예시 문서가 제품의 현재 상태를 정직하게 비추게 한다.
