# 작업 규칙

norii에서 작업하는 사람과 에이전트가 따르는 필수 규칙이다. 세부 근거는 링크된 문서를 단일 출처로 둔다.

## 결정과 보고

- 이 문서 집합에 없는 결정이 필요하거나, 실제 구현이 설계와 달라지면 **임의로 진행하지 말고 사용자에게 먼저 보고**한다.
- **설계 변경 워크플로** — 구현 중 문서와 다른 방향이 필요함을 발견하면:
  1. **멈춘다** — 그 지점에서 해당 부분의 구현을 중단한다. 낡은 설계 위에 코드를 더 쌓지 않는다.
  2. **보고한다** — 문서의 현재 설계 · 발견한 문제 · 대안을 갖춰 보고한다.
  3. **결정 후 문서를 먼저 갱신한다** — 단일 출처가 새 방향의 사양이 된다.
  4. **코드를 정렬한다** — 새 구현뿐 아니라 이미 쌓인 코드도 갱신된 문서에 맞춘다.
- **보고 경계**: 문서가 명시한 규칙·계약·정책과 달라지는 변경만 보고 대상이다. 문서가 규정하지 않은 수준의 구현 세부 선택은 자유다.
- [비목표와 경계 규칙](non-goals.md)의 경계 규칙을 넘는 기능 요청은 별도 제품 결정으로 취급하고 반드시 보고한다.
- 미결정 사항은 [실제 구현 계획](../docs/implementation-plan.md)의 열린 결정 목록에서 추적한다.
- **변경 크기 상한** — 한 검토 단위(보고→승인→커밋)가 **신규·수정 파일 8개 또는 ±300줄을 초과할 것으로 예상되면**, 작업을 시작하기 전에 멈추고 분할 계획(각 단위의 내용·순서)을 먼저 보고한다. 상한 이하의 작업은 한 번에 진행한다. 의도는 검토 가능한 크기 유지다 — 한 번에 파일 수십 개가 쌓이면 사용자가 변경을 실질적으로 검토할 수 없다. **작업 중 실제 변경이 착수 시 보고한 예상을 초과하게 되면**(파일 수 기준), 그 시점에 멈추고 늘어난 사유와 조정된 범위를 보고한 뒤 진행한다 — 시작 전 어림이 빗나가도 중간 체크포인트가 생기게 한다.

## 문서 규칙

- 규칙·설계를 바꿀 때는 코드보다 **문서(단일 출처)를 먼저 갱신**한다.
- **문서 갱신은 완료 조건이다.** 문서가 규정한 동작을 바꾸는 변경은 문서를 **먼저, 늦어도 같은 커밋에서** 갱신한다. 문서 갱신 없이 들어간 동작 변경은 미완성이다.
- **구현 세부는 계약 문서에만 둔다.** 계약 문서란 **앱의 런타임 동작을 규정하는, 코드와 1:1 대응되는 값·시그니처·정책을 담은 문서**다 — 현재 8곳: [Rust 커맨드 계약](../docs/rust-commands.md) · [문서 모델](../docs/document-model.md) · [에디터 전략 — 단축키 계약](../docs/editor-strategy.md#단축키-계약) · [기술 스택 버전 표](../docs/tech-stack.md) · [파일 생명주기 정책](../docs/file-lifecycle.md) · [보안 — CSP](../docs/security.md) · [프리뷰 전략](../docs/preview-strategy.md)(sanitize 정책·플러그인·mermaid securityLevel) · [창 표면 계약](../docs/design/window-chrome.md)(transparent·창 뒤 흐림 반경·폴백). 디자인 값(팔레트·타이포·간격 스케일)은 예외로 문서가 아니라 `panda.config.ts`가 소유한다 — 값을 문서에 복제하지 않는다(→ [디자인 시스템](../docs/design/design-system.md#이-문서의-경계)). 빌드 설정 서술(tsconfig 옵션·경로 별칭)은 제외한다 — 게이트가 실제 설정 파일로 돌므로 문서 드리프트가 동작 차이로 이어지지 않는다. 다른 문서는 링크만 한다. 이 기준을 충족하는 문서가 생기면 목록에 추가한다. 문서-코드 대조의 대상도 이 목록이다. 단 자동 검사(`docs-drift`)는 **기계 대조가 가능한 표면만** 돈다 — 나머지 계약은 마일스톤 종료 시 수동 대조한다(→ [개발 명령](../docs/development-commands.md), [테스트 전략](../docs/testing.md)).
- 각 문서는 한 주제의 단일 출처다. 같은 사실을 여러 문서에 복제하지 않고 링크한다.
- **확정된 디자인 결정은 결정 기록으로 남긴다** — **한 주제에 한 파일**(`.claude/docs/design/decisions/<주제>.md`). 같은 주제가 다시 결정되면 새 파일을 만들지 않고 **그 파일을 고쳐 쓰고**, 무엇이 왜 바뀌었는지만 그 파일의 "바뀐 것" 표에 한 줄 남긴다 — 옛 전문은 git 히스토리가 갖는다. 미결정은 결정 기록이 아니라 열린 결정 목록이 추적한다(→ [결정 기록](../docs/design/decisions/README.md)).
- **문서 링크는 위에서 아래로만 건다.** 루트(`DESIGN.md`)가 상세를 링크하고, 상세는 루트를 링크하지 않는다 — 필요한 사실은 자기가 소유하거나 형제 문서를 가리킨다.
- 새 설계 문서를 추가하면 [AGENTS.md](../../AGENTS.md) 인덱스에 연결한다.

## 코드 규칙

- 책임 경계를 지킨다. 웹뷰는 파일시스템을 `invoke`로만 만지고, Rust는 마크다운 파싱/렌더를 떠안지 않는다(→ [아키텍처](../docs/architecture.md)).
- 새 파일 커맨드는 구현 전에 [Rust 커맨드 계약](../docs/rust-commands.md)에 시그니처를 먼저 추가한다.
- 재사용 로직은 `packages/*`에, 조립·Tauri 연동은 `apps/desktop`에 둔다(→ [파일/폴더 구조](../docs/project-structure.md)).
- 문서 본문은 저장/열기 시점에만 IPC를 건넌다. 키 입력마다 Rust를 호출하지 않는다.

## 품질 게이트

- **TDD 기본**: 신규 기능·커맨드·상태 전이·파서 로직은 **실패하는 테스트를 먼저** 쓰고 구현한다(→ [테스트 전략 · TDD](../docs/testing.md)).
- **테스트는 설명 가능해야 한다**: 새 기능의 테스트는 **왜 필요한지·무엇을 보장하는지·어디까지 커버하는지(경계)**를 드러낸다. 설명 못 하는 테스트는 다시 쓴다(→ [테스트 전략](../docs/testing.md#테스트는-스스로를-설명한다)).
- 커밋 전 `mise run check`를 통과시킨다 — JS/TS(oxfmt·oxlint·Steiger·tsc·Vitest) + Rust(cargo fmt·clippy·test)를 수정 없이 검증한다(→ [코드 품질 전략](../docs/code-quality.md) · [개발 명령](../docs/development-commands.md)).
- 한글 IME 입력은 위험 영역이므로 실제 WKWebView E2E로 자동 검증한다(→ [테스트 전략](../docs/testing.md#위험-영역은-실제-앱으로-검증-핵심) · [에디터 전략](../docs/editor-strategy.md)).
- 자동 검증이 불가능한 영역은 그 이유와 수동 검증 방법을 PR·문서에 남긴다.
- **문서로 시연되는 기능은 예시를 함께 낸다** — 판단은 아래 [예시 문서](#예시-문서-examples)를 따른다.

## 예시 문서 (`examples/`)

`examples/`는 앱으로 열어 눈으로 확인하는 `.md` 모음이다. 자동 테스트가 "동작한다"를 증명한다면, 예시는 **"어떻게 보이는가"** 를 사람이 판정하게 한다. 목록과 작성 규칙은 [examples/README.md](../../examples/README.md)가 소유한다.

**대부분의 기능은 대상이 아니다.** 예시는 눈으로 판정할 대상이 있을 때만 필요하다 — 억지로 만든 예시는 열리지 않고 유지 부담만 남긴다. 다음 두 조건을 **순서대로** 충족해야 대상이다.

```text
1. .md 파일 하나로 시연되는가?
     아니오 → 대상 아님 (파일 열기·저장·탭·사이드바·창 유리·단축키)
2. 사람이 눈으로 판정할 것이 있는가? (타이포·간격·스크롤·깜빡임·실패 표시)
     아니오 → 대상 아님 (통과/실패가 갈리는 것은 테스트의 몫이다)
```

두 조건을 충족해도 **새 파일 생성은 마지막 선택지**다.

```text
기존 예시로 이미 확인된다  → 추가하지 않는다 (대부분 여기에 해당한다)
기존 파일의 주제에 속한다  → 그 파일에 절을 더한다
새로운 표면이다            → 새 파일을 만들고 examples/README.md 목록에 등록한다
```

경계 사례는 셋이다. **에디터 상호작용**(접기·커서·IME)은 문서만으로 시연되지 않아 대상이 아니다 — 다만 수동 검증에 쓸 *재료*(중첩이 깊은 문서 등)가 필요하면 기존 예시에 절을 더한다. **보이지 않는 수정**(성능·메모리·번들)은 대상이 아니며 실측 수치가 그 자리를 대신한다. **시각 토큰 변경**(타이포 스케일·색)도 새 예시를 만들지 않는다 — 기존 문서들이 이미 그 토큰을 쓰므로 그것을 여는 것이 곧 확인이다.

대상이지만 만들지 않기로 판단했다면 **그 근거를 남긴다**(PR·커밋 본문). 위 "자동 검증이 불가능한 영역"과 같은 원칙이다 — 근거가 기록되지 않으면 누락인지 판단인지 구별할 수 없다.

## 성능 규칙

- 번들 크기 목표 < 15MB를 측정·유지한다.
- 커스텀 에디터 데코레이션은 뷰포트 범위에만 적용한다(전체 문서 금지).
- 프리뷰는 디바운스한다. 매 키 입력마다 전체 재파싱하지 않는다.
- **흐림은 스크롤하는 면 위에 겹치지 않는다.** `backdrop-filter`가 스크롤 중인 편집면 위에 걸리면 매 프레임 backdrop이 무효화된다(→ [유리](../docs/design/decisions/glass.md)).
- **창 유리는 idle 비용을 만든다.** 투명 창은 OS의 불투명 fast-path를 포기하므로, 뒤 창이 움직이면 norii가 idle이어도 재합성이 돈다. 창 효과(별도 트랙에서 도입 완료) idle GPU·전력 회귀를 측정하고, 측정 없이 "가볍다"를 주장하지 않는다.
