# 비목표와 경계 규칙

norii의 스코프 판단은 이 문서를 단일 출처로 둔다. 목표(가볍고 빠른 로컬 소스 에디터)만큼 **하지 않는 것**을 명시하는 것이 중요하다. 경계가 흐려지면 성능·단순함이 무너지고, 다른 제품(PKM·협업 툴)으로 미끄러진다.

## 경계 규칙 (한 문장)

> **구조가 텍스트 안(헤딩·들여쓰기·`<details>`)에 살면 채택하고, 텍스트 밖(블록 ID·영구 상태·참조)에 살면 거부한다.**

이 규칙을 넘는 기능은 로컬 `.md` 모델을 깨고 별도 DB(SQLite 인덱스)를 요구한다. 그런 기능이 정말 필요해지면 그것은 **별도의 제품 결정**이며, 임의로 진행하지 않고 사용자에게 보고한다.

## 비목표 목록

| 비목표 | 근거 |
|---|---|
| WYSIWYG / 인라인 리치 편집 | 라운드트립 손실 · IME(한글) 이슈 · 스키마 복잡도. 소스 뷰의 성능·단순함을 우선한다. |
| 블록 아웃라이너 (블록 ID·블록 참조·백링크) | 이 구조는 텍스트 밖 메타데이터를 요구 → DB 필요. Logseq가 파일 모델을 버리고 SQLite로 간 지점. norii는 "에디터"로 남는다. |
| Vault 전체 인덱싱 / 그래프 뷰 / PKM 기능 | 문서 수 폭발 시 파일 모델이 붕괴한다(Logseq: 2,000+ 페이지 = 4~10분 로딩, 300MB↑ 시작 실패). norii는 파일 에디터이지 지식 그래프가 아니다. |
| 실시간 협업 (CRDT/OT) | CRDT + 동기화 서버 + 인프라가 필요해 앱이 무거워진다. 단독 사용자·로컬 파일 모델을 유지한다. |
| 서버 / 클라우드 백엔드 | local-first 원칙. 동기화가 필요하면 사용자의 기존 파일 동기화(iCloud/Dropbox/git)에 위임한다. |

각 근거의 출처는 [참고 사례와 라이선스](prior-art.md)의 Logseq · Obsidian 교훈을 단일 출처로 둔다.

## 경계 안에서 허용하는 것

비목표와 헷갈리기 쉬운, **채택하는** 기능들. 모두 "구조가 텍스트 안에 산다"는 규칙을 통과한다.

- **파일 트리 사이드바** — 폴더/파일의 위계는 파일시스템에 이미 존재한다. 전체 인덱싱이 아니라 단순 트리 표시이므로 안전하다. (→ [문서 모델](../docs/document-model.md))
- **헤딩/리스트 접기(폴딩)** — 접기는 에디터의 표현일 뿐 `.md`를 건드리지 않는다. (→ [에디터 전략](../docs/editor-strategy.md))
- **`<details>` 토글** — 정식 마크다운(HTML 통과)이라 텍스트 안에 남고 다른 에디터에서도 열린다.

## 접힘 상태 영속화의 경계

사용자가 접어둔 상태를 "기억"하려면 그 상태를 어딘가 저장해야 한다. 이때 **`.md` 본문에는 절대 기록하지 않는다** — 넣는 순간 파일이 지저분해지고 호환이 깨진다. 초기에는 저장하지 않고(열 때마다 펼침), 필요해지면 `.md`가 아닌 사이드카 config(예: `.norii/foldstate.json`)에 둔다. 상세는 [파일 생명주기 정책](../docs/file-lifecycle.md).
