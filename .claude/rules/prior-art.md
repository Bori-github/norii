# 참고 사례와 라이선스

norii는 참고 앱을 **라이선스 성격별로** 좁혀 유지한다. 각 앱이 "무엇을, 어디까지 가져올 수 있는가"의 대표 사례다. 이 문서는 그 판단의 단일 출처다.

| 앱 | 라이선스 | 성격 | norii가 취하는 것 |
|---|---|---|---|
| Noteriv | **MIT** ✅ | 코드 차용 가능 | Tauri 2 + React 19 + CM6 — 스택 최근접. 구현을 출처 표기하에 차용한다. |
| VS Code (Code – OSS) | **MIT** ✅ | 코드 차용 가능 (영역 한정) | 파일 처리의 검증된 동작 — 트리 lazy 로딩(`children` 부재=미해석)·자연 정렬·파일 감시(부모 디렉터리 + 삭제 유예 재확인). 출처 표기하에 참고·차용한다. **단 MS 배포 빌드와 아이콘 등 브랜딩 에셋은 비공개 — GitHub의 MIT 코드 저장소만 참고한다.** |
| Logseq | **AGPL** ⚠️ | 개념만 (clean-room) | 접기/아웃라인 UX의 개념. **코드 복사 금지(전염).** |
| Obsidian | 🔒 **비공개** | UX 벤치마크 (clean-room) | 하이브리드 접기·라이브 프리뷰·파일 트리 UX. **코드 없음, 동작만.** |

## 라이선스 규칙

- **MIT / Apache 2.0** → 출처 표기하에 코드 차용 가능 (Noteriv).
- **AGPL** → 개념만 참고. 코드를 복사하면 norii 전체가 AGPL로 전염된다 (Logseq).
- **비공개** → 코드 못 봄. UX·동작 벤치마킹만 (Obsidian).

읽고 배우는 것은 자유지만, **코드 복사·차용**은 위 규칙을 따른다.

> **norii 자체는 MIT**(루트 `LICENSE`, © 2026 bori)로 배포한다. 따라서 MIT·Apache 2.0 의존성은 자유롭게 쓰고, **AGPL 코드 편입만 피하면** 된다(전염 → norii가 AGPL로 강제됨).

## Clean-room 참고 규칙 (Obsidian · Logseq)

Obsidian(비공개)과 Logseq(AGPL)는 **코드를 가져올 수 없다.** 기능·동작·UX를 참고하되, **코드·에셋을 복사하지 않고 관찰한 동작을 공개 사양처럼 삼아 독립 재구현(clean-room)** 한다.

허용하는 것:

```text
- 기능의 동작·UX를 관찰하고 norii 방식으로 독립 재구현
- 상호작용 규칙(단축키 관례, 접기 동작, 사이드바 UX)을 기준점으로 삼기
- 성능·완성도 비교의 벤치마크로 사용
```

하지 않을 것:

```text
- 소스 코드 복사·이식·vendoring
- 플러그인/테마 CSS·에셋·아이콘 복사
- AGPL 코드를 조금이라도 norii에 편입 (전체가 AGPL로 전염됨)
- 독점 마크다운 확장 문법을 그대로 복제해 호환을 표방하는 것
```

판단 기준은 **"코드를 봤는가"가 아니라 "코드에서 가져왔는가"** 다. 동작을 보고 스스로 다시 구현하면 안전하다. 애매하면 [작업 규칙](project-rules.md)에 따라 사용자에게 보고한다.

Noteriv(MIT)는 clean-room 대상이 아니다 — 출처 표기하에 코드를 직접 차용할 수 있다.

## 참고할 기능 (범위 안에서만)

Obsidian·Logseq의 기능 중 **norii의 목표 범위에 드는 것만** 동작 참고 대상으로 삼는다. 두 앱의 대표 기능 상당수(백링크·그래프·블록 참조)는 norii의 [비목표](non-goals.md)이므로 제외한다.

| 기능 | 원본 | norii 범위 | 참고 방식 |
|---|---|---|---|
| 헤딩/리스트 접기 | Obsidian, Logseq | ✅ 채택 | 동작·단축키만 참고, CM6로 독립 구현(→ [에디터 전략](../docs/editor-strategy.md)) |
| 하이브리드 라이브 프리뷰(커서 줄만 원문 노출) | Obsidian | ✅ 향후 후보 | UX만 참고. 현재는 분할 프리뷰(→ [프리뷰 전략](../docs/preview-strategy.md)) |
| 파일 트리 사이드바 | Obsidian | ✅ 채택 | 탐색 UX 참고, 파일시스템 트리로 독립 구현(→ [문서 모델](../docs/document-model.md)) |
| 접이식 불릿 아웃라인 | Logseq | 🔸 제한적 | "리스트 접기" 수준까지만. **블록 아웃라이너(블록 ID·참조)는 비목표** |
| 백링크 · 그래프 뷰 · 블록 참조 | Obsidian, Logseq | ❌ 비목표 | 참고 대상 제외. 텍스트 밖 구조 → DB 필요(→ [비목표](non-goals.md)) |

각 항목의 "참고 방식"이 **동작·UX 참고 + 독립 구현**임을 명시한다.

## 교훈 (설계 결정으로 연결)

### Noteriv — 검증된 스택
Tauri 2 + React 19 + CodeMirror 6 조합이 실제로 동작하는 최근접 사례다. norii의 스택·구조 선택이 안전하다는 증거이며, MIT라 구현을 직접 참고·차용할 수 있다.

### Logseq — 파일 모델의 경계
Logseq는 원래 "마크다운 파일 = 진실"이었으나 2,000+ 페이지에서 4~10분 로딩, 300MB↑ 시작 실패를 겪고 SQLite DB로 전환했다. 교훈: **파일 모델은 PKM 규모에서 붕괴한다.** norii는 "에디터"로 남아 이 벽을 피한다(→ [비목표](non-goals.md)).

### Obsidian — CM6 성능 상한 증명 + Electron 반면교사
Obsidian은 CM6로 대규모 vault를 감당하며 하이브리드 라이브 프리뷰(접기 포함)를 CM6 데코레이션으로 구현한다 → norii의 CM6·접기 선택이 안전하다는 증거. 동시에 Electron 때문에 무겁다는 평은 norii가 Tauri를 고른 이유를 정당화한다.
