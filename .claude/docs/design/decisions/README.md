# 디자인 결정 기록

norii의 **확정된 디자인 결정**을 **한 주제에 한 파일**로 기록한다. "왜 이렇게 정했는가"와 "무엇을 기각했는가"가 여기 산다.

같은 주제가 다시 결정되면 **새 파일을 만들지 않고 그 주제의 파일을 고쳐 쓴다.** 팔레트 하나가 파일 넷에 흩어져 지금 무엇이 유효한지 읽어내기 어려웠던 것이 이 방식의 이유다.

## 세 문서의 경계

같은 사실을 두 곳에 적지 않는다.

```text
/DESIGN.md          지금의 규칙 — 큰 그림 · 불변식 · 표면 표 · 원칙 요약 · 이 목록의 인덱스
decisions/*.md      그 규칙이 나온 이유 — 맥락 · 고려한 대안 · 기각 근거 · 결과
design-system.md    Panda로 어떻게 구현하는가 (계약)
window-chrome.md    창 설정의 실제 값 (계약)
```

DESIGN.md를 읽으면 **무엇을 지켜야 하는지** 알 수 있고, 결정 기록을 읽으면 **왜 그것을 지켜야 하는지** 알 수 있다.

## 규칙

- **확정된 것만 기록한다.** 미결정은 [실제 구현 계획](../../implementation-plan.md#열린-결정-open-decisions)의 열린 결정이 단일 출처다. 두 곳에서 상태를 추적하지 않는다.
- **파일명이 곧 주제다.** 번호를 붙이지 않는다 — 결정은 시간순이 아니라 주제로 찾는다.
- **결정이 뒤집히면 본문을 고쳐 쓰고, "바뀐 것" 표에 한 줄을 남긴다.** 옛 결정의 전문은 지워도 된다 — git 히스토리가 갖는다. 표에 남는 것은 **무엇이 왜 바뀌었는가** 한 줄이며, 이것이 같은 논의의 반복을 막는다.
- **주제가 늘어날 때만 파일을 만든다.** 새 파일은 [템플릿](_template.md)에서 복사해 아래 목록과 루트 `DESIGN.md`의 결정 목록에 등재한다.
- **링크는 루트 → 상세 한 방향이다.** 상세 문서는 루트를 링크하지 않는다 — 필요한 사실은 자기가 소유하거나 형제 상세 문서를 가리킨다.
