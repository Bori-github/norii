# 테스트 전략 · TDD

norii는 **가능한 모든 영역에서 TDD를 기본값**으로 둔다. 이 문서는 테스트 방법론·환경·레이어의 단일 출처다. 도구·버전은 [코드 품질 전략](code-quality.md), 실행 명령은 [개발 명령](development-commands.md)을 단일 출처로 둔다.

## 두 목표를 혼동하지 않는다

- **앱**은 가볍고 빠르게 만든다.
- **테스트**는 가볍게가 목표가 **아니다.** 테스트의 목표는 **운영 환경과의 동일성(fidelity)** 이다. 위험한 영역일수록 테스트는 **운영 환경에 더 가깝게**(실제 앱·실제 웹뷰·실제 IPC로) 한다 — 실행이 느리고 설정 비용이 커도.

> 테스트 환경이 운영 환경과 다르면, 그 차이만큼 버그가 새어 나간다. norii의 운영 환경은 **Tauri 앱 = 실제 Rust 백엔드 + OS 네이티브 웹뷰(mac은 WKWebView) + 실제 IPC**다. 따라서 환경에 의존하는 코드는 **에뮬레이션 DOM(happy-dom)이나 임의 브라우저가 아니라 실제 웹뷰/실제 앱**에서 테스트한다.

환경은 충실도↔비용 순으로 셋이다 — **에뮬레이션 DOM(happy-dom, 금지) < Vitest Browser Mode(WebKit) < 실제 Tauri 앱(tauri-plugin-webdriver)**. 위로 갈수록 운영에 가깝고 실행이 느리다. 어느 대상을 어느 환경에 두는지는 아래 [레이어별 표](#레이어별--무엇을-어디서-테스트하나)가 정한다.

## TDD가 의도하는 것

TDD의 목적은 테스트 개수가 아니라 **책임 경계를 구현 전에 고정**하는 것이다.

```text
Red    → 실패하는 테스트를 먼저 쓴다 (원하는 동작을 고정)
Green  → 그 테스트를 통과하는 최소 구현
Refactor → 테스트를 초록으로 유지하며 정리
```

테스트를 쓰기 전에 묻는다: *이 동작은 사용자의 어떤 경험을 지키는가? 실패하면 어느 책임 영역을 의심하는가?*

## 실패 경로를 먼저 열거한다

"이게 동작하는가"만 물으면 행복 경로에 갇힌다. 이 저장소에서 구현·TDD를 통과하고 머지 직전에야 잡힌 결함들은 전부 "이게 **어떻게 실패하는가**"라는 질문에서만 나왔다 — 취소×실패가 겹친 누수, 거부된 로드의 영구 캐시, 명세 예시에 없던 입력(링크 든 헤딩), 자원 고갈(무한 크기 수식). 기능의 테스트 목록을 쓰기 전에 실패 공간을 먼저 열거한다:

```text
비동기 이펙트    취소 × 실패 × 재시도의 곱집합을 열거한다 — 가드 두 개가 각각 옳아도
                겹치는 상태에서 틀릴 수 있다(누수·이중 처리·영구 고착)
파서·입력 처리   테스트 케이스를 명세의 예시에서 물려받지 않는다 — 입력 공간에서 도출한다
                (비었음 · 중첩 · 마크업 섞임 · 거대 값 · 같은 줄 변형 · 위조 입력)
서드파티 통합    공식 보안 문서를 먼저 읽는다(예: KaTeX security의 maxSize·maxExpand·trust).
                저장소가 그 의존성에 대해 이미 실측한 기록을 검색해 설계 입력으로 잇는다
분기 하나       분기를 썼으면 그 분기를 강제로 태우는 레드 테스트가 있어야 한다 —
                "의도만 있고 실행된 적 없는" 죽은 분기를 남기지 않는다
```

재현이 불가능한 실패 경로(경합 창 등)는 테스트 대신 **그 사유와 수동 검증 방법을 커밋·문서에 남긴다**(→ [작업 규칙 — 품질 게이트](../rules/project-rules.md#품질-게이트)의 "자동 검증이 불가능한 영역"). 재현이 어렵다는 이유로 목록에서 빼지 않는다 — 빼는 순간 그 경로는 아무도 지키지 않는다.

## 테스트는 스스로를 설명한다

새 기능에 붙는 테스트는 **자기 존재를 설명할 수 있어야 한다.** 이름·구조·주석으로 다음 셋이 드러나야 한다.

- **왜 필요한가** — 이 테스트가 지키는 사용자 경험·책임. 실패하면 무엇이 깨지는가.
- **무슨 영향인가** — 어떤 동작·계약을 고정하는가. 이 테스트가 초록이면 무엇이 보장되는가.
- **어디까지 커버하는가(경계)** — 검증하는 범위와, **의도적으로 검증하지 않는 것(경계 밖)**. 커버하지 않는 영역을 분명히 해 "이 테스트가 있으니 안전하다"는 착시를 막는다.

설명할 수 없는 테스트는 **무엇을 지키는지 모르는 테스트**다 — 리팩터링에서 왜 깨졌는지 판단할 수 없고 커버리지 숫자만 올린다. 그런 테스트는 쓰지 않거나, 의도가 서도록 다시 쓴다. 이는 커버리지 원칙(숫자가 아니라 "실패 시 어떤 경험이 깨지는가")의 연장이다.

**계약 문서의 규칙을 검증하는 테스트는 자기가 집행하는 문서 섹션을 인용한다** — 예: `// 집행: file-lifecycle.md#인코딩-정책 — 파이프라인 2단계(널 바이트 규칙)`. 문서가 사양이고 테스트가 집행자다. 동작을 바꾸면 테스트가 깨지고, 테스트를 고치려면 인용된 문서를 마주치게 되어 **코드-문서 간극이 게이트에서 잡힌다**(→ [작업 규칙 — 문서 규칙](../rules/project-rules.md)).

## 레이어별 — 무엇을 어디서 테스트하나

| 대상 | 환경 | 이유 |
|---|---|---|
| 순수 로직 (파서 파이프라인·폴드 범위 계산·상태 리듀서) | Vitest (Node) | **환경 의존이 없음** — 동일 입력→동일 출력이라 Node에서도 운영과 결과가 같다. "가벼워서"가 아니라 "환경 무관이라서" 여기 둔다. |
| 컴포넌트·렌더 (`shared/ui`·`widgets`) | **Vitest Browser Mode (WebKit)** | 실제 렌더·계산된 CSS·측정값이 필요. happy-dom 대체. Playwright WebKit은 WebKit 기반이나 시스템 WKWebView와 동일하진 않다 — 운영 동일 충실도는 실앱 E2E 계층에서만 얻는다. |
| **에디터 · 한글 IME · 데이터 유실 왕복 (위험 영역)** | **tauri-plugin-webdriver (실제 Tauri 앱) + WebdriverIO 클라이언트** | 실제 WKWebView 조합(composition) 이벤트, 실제 Rust 파일 I/O 왕복. **실제 앱 전체를 띄워 — 실행은 가장 느리지만 운영과 동일하게.** 클라이언트는 `webdriverio` programmatic `remote()`를 Vitest 안에서 구동한다(테스트러너 없이 경량). 플러그인은 **`webdriver` Cargo 피처** 뒤에 두어(optional dependency), E2E 실행 시 `tauri dev --features webdriver`로만 컴파일·기동한다 — 일반 개발·릴리스 빌드에는 포함되지 않는다. |
| Rust 백엔드 (커맨드·인코딩·watch) | `cargo test` + 실제 임시 디렉터리 파일 I/O | 백엔드는 이 자체가 운영 동일. |
| IPC 계약 (Rust ↔ 프론트) | **tauri-specta** (Rust→TS 타입 생성) | 직렬화·`AppError` 매핑 드리프트를 **컴파일 타임에 차단**. |

## 위험 영역은 실제 앱으로 검증 (핵심)

norii에서 가장 위험하고 고유한 것 — **한글 IME**와 **데이터 유실 방지(자동 저장·종료 방어·저장/열기 파일 왕복·외부 변경 처리)** — 는 **실제 앱(tauri-plugin-webdriver)에서 끝단까지** 테스트한다. 이들을 유닛·목킹으로 대체하지 않는다. 목킹은 "우리가 맞다고 가정한 것"을 확인할 뿐이라, 실제 웹뷰·IPC에서만 드러나는 문제를 놓치기 쉽다.

예:
- *종료 방어*: 실제 앱에서 문서를 편집 → 곧바로 창 닫기 → 저장 대기분이 플러시돼 디스크에 남는지, Untitled 탭은 실제 다이얼로그가 종료를 막는지.
- *한글 IME*: 실제 WKWebView에서 조합 중 폴딩·자동완성이 오작동하지 않는지.
- *파일 왕복*: 실제 Rust 커맨드로 저장 → 디스크에서 다시 읽어 내용·인코딩·개행이 보존되는지.

**새 사용자 표면은 실앱 E2E 시나리오를 같은 단위에서 함께 낸다.** 단위·위젯 층이 각자 초록이어도 끝단 배선(IPC·이벤트·DOM)은 실앱에서만 증명된다 — 확장 등록 한 줄을 지워도 전 게이트가 통과한 실측 사례가 근거다. 자동화 장벽(네이티브 다이얼로그·수정자 키)을 만나면 "자동화 불가"로 분류하기 전에 **E2E 훅(`expose-e2e-api`)의 기존 패턴으로 우회 가능한지 먼저 확인한다** — 파일·폴더 열기가 이미 이 방식이다. 정말 불가한 것만 수동 검증 대상으로 남기고 그 이유를 기록한다.

## 타이밍 기반 테스트는 반복 실행으로 검증한다

실제 파일시스템 이벤트를 기다리는 테스트(watch 등)는 결과가 타이밍(OS 신호 배달·머신 부하)에 좌우되어 **한 번의 통과가 안정성의 증거가 아니다**. 다음을 규칙으로 한다.

- **작성·수정 시 반복 실행**: "실패할 때까지 N회 반복(예: 15회), 실패 시 그 출력을 통째로 보존"하는 루프로 간헐 실패를 낚는다. 연속 무실패를 확인한 뒤에만 완료로 본다.
- **발견은 코드에 기록한다**: 반복 실행이 드러낸 사실(예: 감시 교체 직후 이전 구독의 늦은 이벤트 배달 — M2 watch에서 실측, 세대 토큰으로 차단)은 해당 코드·테스트 주석에 남기고, 재현·수정 경위는 커밋 본문에 남긴다. 대화·터미널에만 있는 발견은 기록이 아니다.
- **판정 기준은 유출과 정당한 지연을 구별한다**: 시간이 얽힌 단언은 "즉시 상태"가 아니라 시한부 폴링으로 쓰고, 이전에 정당하게 도착한 이벤트를 실패로 오인하지 않게 조건을 좁힌다.
- 반복 통과는 "매우 드묾"의 증거이지 증명이 아니다 — 위험 영역의 최종 확인은 위 절의 실앱 E2E가 맡는다.

## FSD에서의 테스트 배치

- 테스트는 대상 **슬라이스에 co-located**(`*.test.ts`).
- 슬라이스의 **Public API(`index.ts`)를 테스트**한다 — 내부 구현 우회 금지(리팩터링에 강하게).

## CI — 플랫폼별 실제 러너

실제 앱 E2E는 **각 플랫폼 러너에서 그 플랫폼의 실제 웹뷰로** 실행한다 — mac 러너 = 실제 WKWebView, Windows = WebView2, Linux = WebKitGTK. 운영과 동일한 렌더·IPC를 CI가 재현한다(→ [플랫폼 전략](platform-strategy.md)).

## 성숙도 주의

**확인된 한계(0.2.1 실측)**: 이 플러그인은 WebDriver 키 액션의 **수정자 키(Meta/Ctrl)를 매핑하지 않는다** — `Cmd+S` 같은 단축키를 E2E에서 합성할 수 없고, `browser.keys()`의 일반 텍스트도 CM6에 닿지 않는다(**`element.addValue()`만** insertText로 전달됨). 따라서 E2E 시나리오는 동작 트리거를 dev 전용 훅(`window.noriiE2e`)·UI 클릭으로 구성하고, **단축키 계약은 수동 검증 대상**이다 — 검증 방법: `mise run dev`에서 실제 키보드로 [단축키 계약](editor-strategy.md#단축키-계약) 표의 각 항목을 확인한다.

macOS 실앱 E2E 경로는 하나가 아니다 — 공식 임베디드 provider(`@wdio/tauri-service`), CrabNebula 상용 포크(macOS는 유료 키), 그리고 `Choochmeque/tauri-plugin-webdriver`가 있다. norii는 그중 **MIT·무료·임베디드**인 `Choochmeque/tauri-plugin-webdriver`(Rust crate `tauri-plugin-webdriver`, **pre-1.0 커뮤니티** — 버전 핀은 [기술 스택](tech-stack.md#코드-품질)을 단일 출처로 둔다)를 채택한다. 유사명 패키지(`tauri-plugin-webdriver-automation`, `@wdio/tauri-service`)와 혼동하지 않게 정확한 crate·출처를 핀한다. 불안정·정체 시 공식 임베디드 provider나 CrabNebula로 대체하거나 해당 시나리오의 수동 검증 절차와 사유를 기록한다(→ [작업 규칙](../rules/project-rules.md)).

## 게이트·로드맵·커버리지

- **게이트**: `mise run check`가 순수 로직·컴포넌트·Rust 테스트를 실행한다. 실행이 느린 실제 앱 E2E는 CI(및 필요 시 로컬)에서 돈다(→ [코드 품질 — 통합 게이트](code-quality.md#통합-게이트--mise-run-check)).
- **로드맵**: 각 마일스톤 기능은 **실패하는 테스트로 시작**한다(→ [실제 구현 계획](implementation-plan.md)).
- **커버리지**: `@vitest/coverage-v8`로 측정, 초기에는 리포트만(임계값은 코어 안정 후 — 열린 결정). 숫자를 목표로 삼지 않고 "실패 시 어떤 사용자 경험이 깨지는가"를 기준으로 쓴다.
