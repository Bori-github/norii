# 프리뷰 전략 (markdown-it · sanitize · 스크롤 동기화)

norii는 소스 옆에 렌더된 프리뷰를 분할로 보여준다. 이 문서는 프리뷰 파이프라인의 단일 출처다.

핵심 결정은 **파싱/렌더를 웹뷰(JS)에서 한다**는 것이다. Rust로 내리면 라이브 프리뷰가 키 입력마다 IPC 왕복이 되어 반응성이 나빠진다(→ [아키텍처](architecture.md#프리뷰-파서를-웹뷰에-두는-이유)).

## 파이프라인 (웹뷰 내)

```text
소스 텍스트
  → markdown-it   파싱 + HTML 생성 (GFM 옵션: 테이블·체크박스·취소선)
  → DOMPurify     sanitize (필수)
  → 프리뷰 DOM 삽입
```

파이프라인 로직은 `packages/markdown`에 둔다. DOM 삽입은 소비 측(`apps/desktop`)이 담당한다(→ [파일/폴더 구조](project-structure.md)).

**markdown-it 구성**: 테이블·취소선은 기본 프리셋, 오토링크는 `linkify` 옵션 — 단 **`fuzzyLink`는 끈다**. 켜 두면 확장자를 TLD로 착각해 **파일명이 링크가 된다**(`.md`는 몰도바, `.sh`는 세인트헬레나의 실제 도메인이라 `README.md`가 `http://readme.md`로 나간다). 마크다운 문서에 파일명은 늘 나오므로 오탐이 상시적이고, 누르면 OS 브라우저가 엉뚱한 사이트를 연다. 대가로 `www.`로 시작하는 표기의 자동 링크를 잃는다 — 프로토콜을 붙이면 그대로 링크가 된다. 체크박스(작업 목록)는 markdown-it 본체에 없어 **자체 코어 룰(norii-task-list)** 로 구현한다 — 프리뷰는 표시 전용이고 진실은 소스이므로 **disabled 체크박스**로 렌더한다. 스크롤 동기화용 라인 꼬리표(`data-source-line`)는 자체 룰(norii-source-line)이 주입하며, 이때 원시 HTML 토큰에 들어 있는 **위조 꼬리표는 제거**한다(신뢰 경계 — 아래 sanitize 정책).

## 디바운스

매 키 입력마다 전체 재파싱하지 않는다. 변경을 디바운스하고, 여력이 되면 증분 렌더로 발전시킨다. 이는 [작업 규칙](../rules/project-rules.md)의 성능 규칙이다.

**값(M3 실측으로 확정) — 적응형**: 기본 **150ms**에서 시작하고, 직전 렌더 소요에 따라 `clamp(직전 렌더 ms × 3, 150ms, 1000ms)`로 조정한다. 구현 상수·계산은 `apps/desktop/src/widgets/preview-pane/config.ts`가 단일 코드 지점이다.

근거(실측 2026-07-12, Vitest Browser Mode WebKit·macOS 13): 렌더 1회 비용이 2KB 문서 3ms · 61KB 50ms · 246KB 202ms · 620KB 472ms(중앙값). 작은 문서는 150ms면 즉각 반응하고, 렌더가 디바운스보다 오래 걸리는 큰 문서는 타이핑의 짧은 멈춤마다 렌더가 끼어들어 입력이 버벅이므로 렌더 비용에 비례해 간격을 자동으로 벌린다.

## Sanitize는 필수다

DOMPurify는 옵션이 아니라 **필수**다. 마크다운은 원시 HTML을 허용하고, norii는 `<details>` 같은 태그를 의도적으로 통과시킨다(→ [에디터 전략](editor-strategy.md)). 사용자 문서에 스크립트가 섞일 수 있으므로 Tauri 웹뷰라도 sanitize 없이 삽입하지 않는다.

수식(KaTeX)·다이어그램(Mermaid)은 sanitize와 상호작용한다 — KaTeX는 MathML 마크업을 허용해야 하고, Mermaid SVG는 정화 뒤에 렌더한다. 별도 정책은 아래 [수식·다이어그램 지원](#수식다이어그램-지원-채택)에 둔다.

**DOMPurify 정책**: 기본 프로필에 `FORBID_TAGS: ['style']`을 더한다 — 기본 허용되는 `<style>`은 문서의 CSS가 프리뷰 밖 앱 UI 전체를 은폐·위장할 수 있어 차단한다(위협 모델 → [보안](security.md)). `data-*` 속성의 기본 허용은 그대로 둔다 — 라인 꼬리표(`data-source-line`)가 sanitize를 통과하는 근거다. 위조 방어는 두 겹이다: 파서 단계에서 원시 HTML 토큰의 위조 꼬리표를 제거하고, 수집 단계(`collectLineBlocks`)에서 유한한 1 이상·비내림차순 값만 수용한다(이진 탐색의 정렬 전제 보호). 프리뷰 패널은 CSS 격리(`contain: paint`)로 문서 인라인 스타일(position:fixed 등)이 패널 밖에 그려지는 것도 막는다.

## 링크 정책

프리뷰 패널이 링크 클릭을 처리하는 방식이다. **허용 스킴 집합과 그 근거는 [보안 — 외부 링크](security.md#4-외부-링크-프리뷰에서-문서-밖으로-나가는-유일한-통로)** 를 단일 출처로 둔다 — 여기서는 프리뷰의 동작만 정한다.

클릭은 **항상 가로채 웹뷰 내비게이션을 막는다** — 링크의 종류와 무관하다. 앱 창이 문서 속 URL로 이동하면 앱 UI가 사라지고 원격 페이지가 그 자리를 차지한다. 가로챈 뒤 **세 갈래**로 나눈다.

```text
문서 밖으로 나가는 링크   http · https · mailto → plugin-opener로 OS 기본 브라우저
문서 안에서 끝나는 링크   #앵커                 → 프리뷰를 그 위치로 스크롤 (앱이 직접 해석)
그 외                    상대 경로·기타 스킴    → 조용한 무동작
```

- **`#앵커`는 외부 링크가 아니다.** 브라우저로 넘길 대상이 아니라 앱이 내부에서 해석해야 할 링크이므로, 스킴 허용목록 판정 **이전에** 갈라진다. 각주(`#fn1`·`#fnref1`)와 헤딩(아래 [헤딩 앵커](#헤딩-앵커))이 이 경로를 탄다.
- **앵커 이동은 에코 가드를 경유하지 않는다.** 가드는 "이 스크롤은 발행하지 마라"는 표시이므로, 경유하면 **에디터가 따라오지 않는다**. 앵커 클릭은 사용자가 일으킨 이동이니 평범한 스크롤로 적용하고, 그 결과 나는 scroll 이벤트가 평소대로 발행되어 에디터가 따라온다. 왕복 루프는 **수신 측**(에디터)이 이미 가드를 쓰기 때문에 생기지 않는다(→ [스크롤 동기화](#스크롤-동기화)).
- **앵커 대상은 프리뷰 콘텐츠 안에서만 찾는다.** 문서 전체(`document.getElementById`)에서 찾으면 프리뷰 밖 앱 UI의 같은 `id`를 잡을 수 있다. `href`는 퍼센트 인코딩될 수 있으므로 디코딩 후 선택자를 만든다.
- 대상 `id`가 문서에 없으면(끊긴 앵커) **무동작**이다 — 에러를 띄우지 않는다.
- 허용되지 않은 링크는 **조용한 무동작**이다 — 에러 배너를 띄우지 않는다. 문서를 열었을 뿐인 사용자에게 보안 경고를 들이밀지 않는다.
- **상대 경로로 다른 `.md`를 여는 것은 아직 무동작**이다 — 허용 루트 밖 경로 차단·이미 열린 탭 전환·미저장 문서의 기준 디렉터리 부재를 함께 풀어야 하므로 [열린 결정](implementation-plan.md#열린-결정-open-decisions)으로 남는다.

## 스크롤 동기화

소스↔프리뷰 스크롤을 연동한다. 소스 라인 ↔ 렌더 블록 매핑 테이블을 유지한다(`packages/markdown`). 매핑·연동 로직은 norii가 직접 구현한다.

정렬 규칙은 **"맨 윗줄 맞추기"(블록 단위 근사) + 가장자리 스냅**이다 — 같은 내용이라도 양쪽 높이가 달라(제목·표 등) 한쪽이 바닥에 닿아도 반대쪽 바닥이 어긋나므로, 바닥에 닿으면 반대 패널도 바닥으로 스냅한다(맨 위는 0=0이라 자연히 맞는다).

## 두 파서 원칙

에디터(CM6 내장 Lezer)와 프리뷰(markdown-it)는 별개 파서다. **완벽 일치를 추구하지 않는다.** 하이라이팅은 근사치여도 되고, 아주 드문 중첩 문법에서 둘의 해석이 미묘하게 갈릴 수 있다. **최종 결과물의 진실은 프리뷰(markdown-it)** 다. 이 원칙을 잡으면 두 파서를 억지로 맞추느라 복잡해지는 것을 피한다.

## 수식·다이어그램 지원 (채택)

각주·수식(KaTeX)·다이어그램(Mermaid)을 **지원한다.** 도입은 이후 마일스톤이다(→ [실제 구현 계획](implementation-plan.md)). 셋은 구현 방식·번들·sanitize 경계가 서로 달라 아래에 못박는다.

```text
markdown-it-footnote        각주        markdown-it 플러그인
@vscode/markdown-it-katex   수식(KaTeX) markdown-it 플러그인 (Microsoft 포크)
mermaid                     다이어그램  플러그인 아님 — fence 렌더 + 클라이언트 렌더
```

### 수식 (KaTeX)

- **`@vscode/markdown-it-katex`**(Microsoft 유지보수 포크)를 쓴다. 원본 `markdown-it-katex`는 사실상 대체됐고 `markdown-it-texmath`도 저활동이라 낮은 bus-factor다. VS Code가 실사용하는 1st-party 포크가 안전하다.
- 인라인 `$…$`·블록 `$$…$$`. KaTeX는 HTML/MathML을 출력하므로 **sanitize가 그 마크업을 깎지 않도록** DOMPurify 정책에 `<semantics>`·`<annotation>`(과 그 `encoding` 속성)을 허용한다 — 이 둘은 DOMPurify의 MathML 허용목록에 없어, 그냥 두면 껍데기만 벗겨지고 안의 TeX 원문이 맨 텍스트로 남아 수식이 두 번 읽힌다(→ [Sanitize는 필수다](#sanitize는-필수다)).
- **katex는 ESM 빌드로 고정한다** — 수식 플러그인은 내부에서 `require("katex")`로 CJS 빌드를 잡는데, 그 빌드가 번들되면 함수 등록이 날아가 **모든 제어 시퀀스가 "정의되지 않음"이 된다**(`\frac`·`\int` … 수식 전체가 죽는다, M4 실측). Vite/Vitest 설정에서 `katex → katex/dist/katex.mjs` alias로 끊는다. 이 방어는 **패키지·앱 양쪽 설정에 다 걸려야** 한다.
- KaTeX CSS·폰트는 **로컬 번들**(외부 CDN 금지 — CSP `style-src`·`font-src 'self'`, → [보안](security.md)).
- **사용자 지정 크기는 `maxSize: 100`(em)에서 자른다** — 기본값 Infinity로 두면 `\rule{9999em}{9999em}` 하나로 문서 높이가 수백만 px이 된다(레이아웃·스크롤 동기화 폭파). 문서는 못 믿는 입력이다.
- 조판 실패(문법 오류)는 그 수식 자리에만 흐린 글자로 남는다 — 배너를 띄우지 않는다(다이어그램과 같은 원칙).

### 각주

- **`markdown-it-footnote`**. 본문의 위첨자 참조와 문서 끝의 각주 목록이 서로를 가리키는 문서 내 앵커(`#fn1`·`#fnref1`)로 이어진다 — 그 `id`·`href`가 sanitize를 통과하는 것이 각주가 동작하는 조건이다.
- **참조와 목록을 오갈 수 있다** — 위첨자 참조를 누르면 각주 목록으로, 되돌아가기 화살표(↩)를 누르면 참조 자리로 스크롤한다. 이 이동은 [링크 정책](#링크-정책)의 `#앵커` 갈래를 탄다(브라우저로 나가지 않는다).

### 다이어그램 (Mermaid)

- Mermaid는 **markdown-it 플러그인이 아니다.** ` ```mermaid ` 펜스를 커스텀 fence 규칙으로 **빈 플레이스홀더**(`<div class="norii-mermaid" data-mermaid-source="…">`)로 내보내고, DOM 삽입 후 클라이언트에서 `mermaid`가 SVG로 렌더한다. 두 이름(클래스·속성)이 파서(`packages/markdown/src/mermaid.ts`)와 소비 측(`apps/desktop`의 `use-mermaid`) 사이의 계약이다.
- **원문은 퍼센트 인코딩해 싣는다.** DOMPurify는 속성값에 `-->`(XML 주석 닫기)가 들어 있으면 그 속성을 통째로 버리는데(SAFE_FOR_XML 방어), `-->`는 플로차트의 기본 화살표다 — 날것으로 실으면 다이어그램이 조용히 사라진다. 인코딩·디코딩은 `mermaid.ts`가 함께 소유한다.
- **sanitize 순서**: mermaid SVG는 신뢰된 엔진 출력이므로 원문을 **DOMPurify로 정화한 뒤** 그 자리에 렌더하고, mermaid `securityLevel: 'strict'`로 스크립트를 봉쇄한다. (사용자 원문은 sanitize를 거치고, SVG는 엔진이 안전한 자리에만 그린다.)
- **기동 비용 방어**: mermaid는 무겁다(d3 등 동반). **문서에 다이어그램이 있을 때만 동적 import(lazy-load)** 한다. 이것은 **번들 크기 대책이 아니다** — 분할된 청크도 앱에 함께 실리므로 예산에는 그대로 잡힌다. 지키는 것은 다이어그램 없는 문서의 기동·메모리 비용이다. 예산은 실측이 따로 답한다 — M4 실측(→ [플랫폼 전략 — 번들 크기 측정](platform-strategy.md#번들-크기-측정)의 실측 기록)으로 mermaid를 포함해도 예산 안임을 확인했고, 그래서 범위를 줄이지 않는다.
- **라이브 프리뷰 비용**: 디바운스 갱신마다 전체 다이어그램을 다시 그리지 않는다. (테마, 원문)을 키로 SVG를 캐시해 **내용이 바뀐 펜스만** 다시 그린다. 캐시에는 **상한(32개, 오래된 것부터 폐기)** 을 둔다 — 다이어그램을 타이핑하는 동안 갱신마다 편집 중간 상태가 새 키가 되므로, 상한이 없으면 SVG가 무한히 쌓인다.
- **렌더 실패**: 문법 오류는 사용자가 고칠 수 있는 일상이라 에러 배너로 올리지 않고 **그 다이어그램 자리에만** 알린다. 실패는 캐시하지 않는다(고치는 즉시 다시 시도한다).

### 헤딩 앵커

목차 링크(`[개요](#개요)`)가 동작하려면 헤딩에 `id`가 있어야 한다. markdown-it은 헤딩에 `id`를 붙이지 않으므로 **자체 룰로 slug를 부여**한다 — 이것이 없으면 `#앵커` 이동은 각주에서만 동작하고 목차 링크는 죽은 채로 남는다.

**slug 규칙(GitHub 호환)** — 사용자가 GitHub에서 쓰던 목차가 norii에서도 그대로 동작해야 한다.

```text
소문자화 → 공백은 하이픈(-) → 영숫자·하이픈·밑줄·유니코드 글자만 남기고 제거
중복이면 뒤에 -1, -2 …  (같은 제목이 두 번 나오는 문서가 흔하다)

"개요"          → #개요
"API 설계"      → #api-설계
"Step 1: 준비"  → #step-1-준비
```

한글은 그대로 남긴다(제거하면 한글 제목의 앵커가 전부 빈 문자열이 된다). 클릭 시 `href`의 퍼센트 인코딩을 디코딩해 대조한다.

### 콜아웃 (GFM alerts)

인용문 안의 첫 줄이 `[!NOTE]` 꼴이면 강조 상자로 렌더한다.

```text
> [!WARNING]
> 이 작업은 되돌릴 수 없습니다.
```

- **GitHub 표준(GFM alerts) 5종만** 지원한다 — `NOTE` · `TIP` · `IMPORTANT` · `WARNING` · `CAUTION`. Obsidian의 콜아웃은 종류가 더 많고 접기(`> [!NOTE]-`)까지 있는 독자 확장이라 채택하지 않는다([참고 사례 — clean-room](../rules/prior-art.md#clean-room-참고-규칙-obsidian--logseq)의 "독점 문법 복제 금지").
- **문법이 이미 마크다운이다.** `>`는 인용문이고 `[!WARNING]`은 그 안의 텍스트다 — norii가 렌더하지 않아도 다른 에디터에서 평범한 인용문으로 읽힌다. 호환이 깨지지 않는다.
- **5종 밖의 타입(`> [!FOO]`)은 평범한 인용문으로 남는다** — 임의로 기본 상자를 씌우지 않는다. 씌우면 다른 에디터와 결과가 갈린다.
- 구현은 markdown-it 코어 룰(인용문 토큰의 첫 문단 검사)로 하고, 상자의 클래스 이름이 sanitize를 통과해야 한다(→ [Sanitize는 필수다](#sanitize는-필수다)).
- **아이콘은 마크업이 아니라 CSS로 그린다**(`::before`). SVG를 마크업으로 넣으면 그만큼 sanitize 허용 표면이 늘고, 문서가 위조한 콜아웃도 아이콘을 갖게 된다. 클래스만 통과시키면 표면이 늘지 않는다.
- **색으로 구별하지 않는다(현재).** 이 디자인 시스템에는 상태색(경고 노랑·위험 빨강 …)이 아직 없다 — 상태색 도입은 탭 ⚠ 배지·에러 배너와 함께 정할 [열린 결정](implementation-plan.md#열린-결정-open-decisions)이고, 유리 위 대비를 다시 계산해야 하는 별도 트랙이다. 콜아웃 하나 때문에 급히 팔레트를 만들면 나중에 배지·배너가 그 색에 끌려간다.

  그래서 5종은 **아이콘 · 라벨 · 왼쪽 굵은 경계선**으로 구별하고, 색은 기존 토큰(`border`·`bg.hover`·`text`·`text.muted`·`accent`)만 쓴다. 상자의 값어치는 "눈에 띈다"이지 "노랗다"가 아니다. 타입별 클래스(`norii-callout-warning`)가 이미 붙으므로, 상태색이 확정되면 **CSS만 바꾸면 된다.**

### 코드 복사 버튼

프리뷰의 코드 블록에 마우스를 올리면 오른쪽 위에 복사 버튼이 뜨고, 누르면 코드가 클립보드에 들어간다.

- **문서를 바꾸지 않는다.** 버튼은 파서가 아니라 소비 측이 프리뷰 DOM에 붙이는 UI다 — `.md`에는 아무것도 남지 않는다.
- 버튼은 **코드 블록에 마우스를 올렸을 때만** 나타난다 — 읽는 동안에는 화면에 없다.
- **프리뷰가 다시 렌더될 때마다 버튼은 사라진다** — 내용을 `innerHTML`로 통째로 교체하기 때문이다(→ [프리뷰 DOM은 React가 소유하지 않는다](#프리뷰-dom은-react가-소유하지-않는다)). 다이어그램 SVG와 같은 경로로 **삽입 뒤 다시 붙인다.** 버튼은 React 컴포넌트이고, 삽입 뒤 코드 블록을 다시 수집해 **포털로 꽂는다** — 비 React 내용 속에서 버튼만 예외로 React가 소유한다(아이콘·복사됨 상태·타이머 관리).
- **코드 블록마다 네이티브 리스너를 달지 않는다** — 블록 수만큼 리스너가 생기고 매 렌더마다 다시 등록되기 때문이다. 포털 버튼의 클릭은 React 합성 이벤트(루트 위임 하나)를 탄다.
- 복사할 원문은 렌더된 텍스트가 아니라 **코드 블록의 원문**이다.
- 버튼은 우리가 넣는 요소이므로 스크롤 동기화의 라인 꼬리표(`data-source-line`)를 달지 않는다 — 매핑 테이블을 오염시키지 않기 위해서다(→ [스크롤 동기화](#스크롤-동기화)).
- **클립보드는 웹 표준 API를 쓰되 `execCommand("copy")` 폴백을 둔다** — Tauri 플러그인은 쓰지 않는다(권한·번들 증가 없음).

  실측(2026-07-14, 실앱 WKWebView·개발 모드): `isSecureContext=true`, `navigator.clipboard.writeText` 성공, `execCommand("copy")` 성공. 다만 **개발 모드는 `http://localhost`에서 서빙되어 항상 secure context**이고, 배포 빌드는 커스텀 프로토콜(`tauri://localhost`)이라 origin이 다르다 — 릴리스 빌드에는 WebDriver 피처가 없어 같은 방식으로 자동 검증할 수 없다. 폴백을 두는 이유가 이것이다: secure context가 아니어도 복사는 동작한다.

  **수동 검증 대상**: 서명된 배포 빌드에서 복사 버튼이 실제로 동작하는지 확인한다(자동 검증 불가 → [작업 규칙 — 품질 게이트](../rules/project-rules.md#품질-게이트)).

### 프리뷰 DOM은 React가 소유하지 않는다

프리뷰 내용은 `dangerouslySetInnerHTML`이 아니라 **위젯이 이펙트에서 직접 `innerHTML`로 넣는다.**

React는 `dangerouslySetInnerHTML` 요소의 자식을 **재렌더마다 다시 만든다** — HTML 문자열이 그대로여도, DOM을 건드리지 않아도, 값을 메모이즈해도 그렇다(M4 실측). 그러면 클라이언트가 직접 꽂은 SVG(다이어그램)가 **아무 재렌더에나 조용히 사라진다.** 내용의 소유권을 위젯이 가져오면 React는 그 자식들을 건드리지 않고, 갱신은 렌더된 HTML이 실제로 바뀔 때만 일어난다.

삽입되는 것은 언제나 sanitize를 마친 HTML뿐이라는 계약은 그대로다(→ [Sanitize는 필수다](#sanitize는-필수다)).
